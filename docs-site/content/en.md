# cppkg-cli

<p align="center">
  <img src="./assets/icon.png" alt="cppkg-cli icon" width="128" height="128">
</p>

[Chinese](./zh-CN.html)

`cppkg-cli` is a project-local C/C++ dependency downloader. It does not install libraries into system directories. Instead, it keeps downloaded dependencies inside the current project so the install layout can be committed, reproduced, inspected, and cleaned up.

There are two install outcomes:

| Dependency type | Default location | Typical examples |
| --- | --- | --- |
| Reusable headers | `cpp_libs/include/` | `nlohmann/json`, `fmt` |
| Full source projects | `cpp_libs/projects/` | GUI libraries, SDKs, projects with `CMakeLists.txt` |

For exact command options, run `cppkg-cli <command> --help`.

## Install

```bash
npm install -g cppkg-cli
```

## Fast Check

```bash
cppkg-cli init
cppkg-cli add nlohmann/json --install
cppkg-cli compile src/main.cpp -o app
```

This is the shortest path for trying cppkg in a new project:

| Step | Command | Result |
| --- | --- | --- |
| 1 | `cppkg-cli init` | Creates `cppkg.json`. |
| 2 | `cppkg-cli add nlohmann/json --install` | Writes the dependency and installs headers. |
| 3 | `cppkg-cli compile src/main.cpp -o app` | Compiles with `cpp_libs/include`. |

If your source contains:

```cpp
#include <nlohmann/json.hpp>
```

then `cppkg-cli compile` automatically adds the default `./cpp_libs/include` path after installation.

## Pick A Command First

Start with the command that matches your workflow:

| What you want | Command | Changes `cppkg.json` |
| --- | --- | --- |
| Try a package temporarily | `cppkg-cli get <url>` | No |
| List installed packages | `cppkg-cli list` | No |
| Add a long-term project dependency | `cppkg-cli add <source> --install` | Yes |
| Install everything from `cppkg.json` | `cppkg-cli install` | No |
| Search before you know the repository URL | `cppkg-cli search <keywords>` | No |
| Manage packages in a browser | `cppkg-cli server` | Optional |
| Compile one or a few source files | `cppkg-cli compile <files...>` | No |

For project dependencies, prefer `add` plus `install`. Use `get` for temporary experiments and validation.

## Browser Web Server

Start the local package manager when you want to browse packages, edit config, and watch install logs from a web page:

```bash
cppkg-cli server
```

By default, the server binds to `127.0.0.1:4936` and prints the URL to open. Pick another port or bind interface when needed:

```bash
cppkg-cli server --port 0
cppkg-cli server --host 0.0.0.0 --port 4936
```

The web UI includes:

| Tab | What it does |
| --- | --- |
| Installed packages | Shows installed package metadata from `cpp_libs/deps.json`. |
| Search and download | Searches GitHub repositories and queues downloads or manifest installs. |
| Direct download | Accepts repository URLs, `owner/repo`, Gitee URLs, and zip URLs. Pasting a GitHub/Gitee release or branch URL fills the package name plus `tag` or `branch`. |
| Manifest | Shows dependencies from `cppkg.json` and any manifest parse error. |
| Status | Checks manifest, lockfile, installed metadata, and tracked files for project health issues. |
| Config | Reads and writes `cppkg.config.json`; token values are redacted in the browser. |
| Tasks | Shows queued/running/completed tasks with live logs and lets you cancel queued work. |

Download and manifest actions run as background tasks, so the page can keep showing progress while a package is being resolved, downloaded, extracted, and recorded.

When developing this repository, use the Vite-powered web UI:

```bash
npm run server:web:dev
```

That script starts the cppkg API server and the Vite UI together. If the default API port is busy, it chooses another local port and wires the Vite `/api` proxy to it.

## Where Files Go

By default, cppkg only writes inside the current project:

| Path | Purpose |
| --- | --- |
| `cppkg.json` | Project dependency manifest. |
| `cppkg-lock.json` | Resolved archive URLs and SHA-256 integrity data. |
| `cppkg.config.json` | Optional project config such as proxy, tokens, and install directories. |
| `cppkg-toolchains.json` | Optional compiler profile config. |
| `cppkg.cmake` | CMake integration file generated by `cppkg-cli cmake`. |
| `cpp_libs/include/` | Shared include directory for reusable headers. |
| `cpp_libs/projects/` | Full source projects. |
| `cpp_libs/cache/` | Downloaded archive cache. |
| `cpp_libs/deps.json` | Installed package metadata used by `list`, `status`, `update`, and `remove`. |

Typical layout:

```text
your-project/
├── cppkg.json
├── cppkg.cmake
├── cppkg-lock.json
├── cppkg-toolchains.json
└── cpp_libs/
    ├── deps.json
    ├── cache/
    ├── include/
    │   ├── nlohmann/
    │   │   └── json.hpp
    │   └── fmt/
    │       └── format.h
    └── projects/
        ├── lvgl_lvgl/
        └── mirrors_jsoncpp/
```

## Common Workflows

### Manage Dependencies With A Manifest

Use this when the dependency belongs to the project and should be installed by other developers or CI.

```bash
cppkg-cli init
cppkg-cli add nlohmann/json --install
```

Common variants:

```bash
cppkg-cli add fmtlib/fmt --name fmt --tag 11.2.0 --install
cppkg-cli add gitee.com/mirrors/jsoncpp --full-project --install
```

Result:

- The dependency is written to `cppkg.json`.
- The installed result is recorded in `cpp_libs/deps.json`.
- Resolved archive and integrity data is written to `cppkg-lock.json`.

Install all dependencies later on another machine or in CI:

```bash
cppkg-cli install
```

Install only selected dependencies:

```bash
cppkg-cli install json fmt
```

Require the lockfile to match the manifest:

```bash
cppkg-cli install --frozen-lockfile
```

### Install A Source Temporarily

Use this when you only want to verify a package and do not want to write it into the project manifest.

```bash
cppkg-cli get https://github.com/nlohmann/json
```

Install multiple sources at once:

```bash
cppkg-cli get https://github.com/nlohmann/json https://github.com/fmtlib/fmt
```

Install a zip archive and point cppkg at its include directory:

```bash
cppkg-cli get https://example.com/vendor.zip --include-path include
```

Result: `get` installs the source and records it in `cpp_libs/deps.json`, but it does not modify `cppkg.json`. If the dependency becomes permanent, use `add <source> --install`.

Supported sources:

| Source | Example |
| --- | --- |
| GitHub repository | `https://github.com/nlohmann/json` |
| GitHub API repository | `https://api.github.com/repos/nlohmann/json` |
| Gitee repository | `https://gitee.com/mirrors/jsoncpp.git` |
| Gitee API repository | `https://gitee.com/api/v5/repos/mirrors/jsoncpp` |
| Remote zip archive | `https://example.com/downloads/my-sdk.zip` |

Common install options:

| Need | Option example |
| --- | --- |
| Pin a release tag | `--tag v3.12.0` |
| Use a branch | `--branch master` |
| Use a semantic version range | `--version-range '^1.2.0'` |
| Use the default branch directly | `--version-policy default-branch` |
| Allow prerelease versions | `--prerelease` |
| Force full-project install | `--full-project` |
| Bypass cache and download again | `--no-cache` |
| Select an include directory inside a zip | `--include-path include` |
| Select the source root inside a zip | `--strip-prefix sdk` |
| Copy only top-level entries | `--components vendor` |
| Verify the downloaded archive | `--checksum sha256:<digest>` |

### Search And Install

Use this when you know what kind of library you need, but not the exact repository URL.

```bash
cppkg-cli search json
cppkg-cli search http client --limit 20
cppkg-cli search gui --language C++
```

By default, search uses `language:C++`, excludes forks and archived repositories, and sorts by star count.

In an interactive terminal, search opens a selector below the results:

| Key | Action |
| --- | --- |
| Up/Down | Move selection |
| Enter | Install the selected repository |
| `q` or Esc | Cancel |

In scripts or CI, install by result index:

```bash
cppkg-cli search json --select 1
```

Only print results and skip the selector:

```bash
cppkg-cli search json --no-interactive
```

Result: installing from `search` behaves like `get`; it does not write to `cppkg.json`.

### Infer Dependencies From Includes

Use this when source files already contain `#include <...>` directives and you want to know what is missing.

```bash
cppkg-cli inspect
cppkg-cli inspect --add
cppkg-cli inspect --install
```

| Command | Result |
| --- | --- |
| `cppkg-cli inspect` | Only reports missing dependencies. |
| `cppkg-cli inspect --add` | Writes recommended dependencies to `cppkg.json`. |
| `cppkg-cli inspect --install` | Writes them to the manifest and installs them. |

Common include patterns such as `fmt/*` and `nlohmann/*` are mapped to installable package sources when possible.

## Compile And Build

Pick the command based on project type:

| Project type | Recommended command | Meaning |
| --- | --- | --- |
| Single file or a few source files | `cppkg-cli compile <files...>` | Calls the compiler directly and adds cppkg include paths. |
| CMake project | `cppkg-cli build` | Runs the CMake configure/build workflow. |
| Inspect the generated command first | `--dry-run` | Prints commands without compiling or building. |
| Use a consistent compiler environment | `--docker` or `--toolchain` | Uses a Docker image or a saved compiler profile. |

### Compile Source Files Directly

For small projects without CMake, use `compile`. It adds the configured cppkg include directory to the compiler arguments:

```bash
cppkg-cli compile src/main.cpp -o app
cppkg-cli compile src/main.cpp src/app.cpp --compiler clang++ --std c++23 -o build/app
```

Common options:

| Need | Example |
| --- | --- |
| Set output file | `-o app` |
| Set compiler | `--compiler clang++` |
| Set C++ standard | `--std c++23` |
| Use a toolchain profile | `--toolchain gcc-14` |
| Compile in Docker | `--docker --docker-image gcc:latest` |

### Build CMake Projects

For CMake projects, use `build`. The default build directory is `./build`:

```bash
cppkg-cli build
cppkg-cli build --release
cppkg-cli build --target app --build-dir cmake-build
```

If `cppkg.cmake` is missing, `build` generates it and injects the cppkg include path during CMake configure.

Common options:

| Need | Example |
| --- | --- |
| Release build | `--release` |
| Set target | `--target app` |
| Set build directory | `--build-dir cmake-build` |
| Use a toolchain profile | `--toolchain project-clang` |
| Build in Docker | `--docker --docker-image cppkg-build:latest` |

### Dry Run

```bash
cppkg-cli compile src/main.cpp --dry-run
cppkg-cli build --release --dry-run
```

`--dry-run` is useful when you want to inspect the compiler or CMake command without producing build outputs.

### Docker Compiler Environment

```bash
cppkg-cli compile src/main.cpp -o app --docker --docker-image gcc:latest
cppkg-cli build --docker --docker-image cppkg-build:latest
```

The default Docker image for `compile` is `gcc:latest`. `build` requires an image that also contains `cmake`.

### Manage Compiler Profiles

Compiler profiles are stored in `cppkg-toolchains.json`. Built-in Docker profiles include `gcc-13`, `gcc-14`, `gcc-latest`, `clang-17`, `clang-18`, and `clang-latest`.

```bash
cppkg-cli compiler list
cppkg-cli compiler install clang-18 --set-default
cppkg-cli compiler current
cppkg-cli compiler use gcc-14
```

Create a project profile:

```bash
cppkg-cli compiler add project-clang \
  --kind clang \
  --compiler-version 18 \
  --docker-image cppkg-clang:18 \
  --set-default
```

Use a saved profile explicitly, or rely on the default profile:

```bash
cppkg-cli compile src/main.cpp --toolchain gcc-14 -o app
cppkg-cli build --toolchain project-clang
```

## Maintenance

List installed packages:

```bash
cppkg-cli list
```

Check manifest, lockfile, metadata, and installed files:

```bash
cppkg-cli status
cppkg-cli doctor
```

Update all packages or one package:

```bash
cppkg-cli update
cppkg-cli update json
cppkg-cli update json --tag v3.12.0
cppkg-cli update lvgl --branch master
cppkg-cli update lvgl --full-project
```

Remove one package:

```bash
cppkg-cli remove json
```

Manage the archive cache:

```bash
cppkg-cli cache list
cppkg-cli cache clean
cppkg-cli cache clean --older-than 30
```

Selectors supported by `install`, `update`, and `remove`:

| Selector | Example |
| --- | --- |
| Manifest dependency name or installed package name | `json` |
| Repository path | `/nlohmann/json` |
| `owner/repo` | `nlohmann/json` |
| Host shorthand | `github.com/nlohmann/json`, `gitee.com/mirrors/jsoncpp` |
| Recorded source URL | `https://github.com/nlohmann/json` |

`install` selectors match entries in `cppkg.json`. `update` and `remove` selectors match installed records in `deps.json`.

## cppkg.json Format

`cppkg.json` is the project dependency manifest. Prefer generating it with `add` instead of writing it by hand:

```bash
cppkg-cli add nlohmann/json
cppkg-cli add fmtlib/fmt --name fmt --tag 11.2.0
cppkg-cli add gitee.com/mirrors/jsoncpp --full-project
```

The three most common forms:

| Need | Form |
| --- | --- |
| Declare only the source | `"json": "https://github.com/nlohmann/json"` |
| Pin a version | `"fmt": { "source": "...", "tag": "11.2.0" }` |
| Force full-project install | `"lvgl": { "source": "...", "branch": "master", "fullProject": true }` |

The recommended syntax is an object map. The map key is the dependency name and the selector you usually use later with `install`, `update`, or `remove`:

```json
{
  "dependencies": {
    "json": "https://github.com/nlohmann/json",
    "fmt": {
      "source": "https://github.com/fmtlib/fmt",
      "tag": "11.2.0"
    },
    "lvgl": {
      "source": "https://github.com/lvgl/lvgl",
      "branch": "master",
      "fullProject": true
    }
  }
}
```

Array syntax is also supported. For array entries, set `name` explicitly when you want a clear selector later:

```json
{
  "dependencies": [
    "https://github.com/nlohmann/json",
    {
      "name": "lvgl",
      "source": "https://github.com/lvgl/lvgl",
      "branch": "master",
      "fullProject": true
    }
  ]
}
```

Note: `add` accepts `owner/repo`, `github.com/owner/repo`, or `gitee.com/owner/repo`. In handwritten `cppkg.json`, `source` must be a full `http` or `https` URL.

### Source And Version Fields

| Field | Type | Meaning |
| --- | --- | --- |
| `source` | string | GitHub repository URL, GitHub API repository URL, Gitee repository URL, Gitee API repository URL, or remote zip URL. |
| `name` | string | Optional selector name for array entries. In object-map syntax, the map key is the dependency name. |
| `tag` | string | Install a specific release tag. If no matching release exists, install the matching repository tag archive. |
| `branch` | string | Install a specific repository branch. |
| `versionRange` | string | Install the highest release tag matching a semantic version range, such as `^1.2.0`, `~1.2.0`, `1.x`, or `>=1.0.0 <2.0.0`. |
| `versionPolicy` | string | Version policy: `latest-release`, `latest-prerelease`, or `default-branch`. |
| `prerelease` | boolean | Allow prereleases when resolving latest release or version ranges. |

Each dependency can choose only one version source: `tag`, `branch`, `versionRange`, and `versionPolicy` cannot be combined.

### Install Modifier Fields

| Field | Type | Meaning |
| --- | --- | --- |
| `fullProject` | boolean | Skip include detection and install as a full project. |
| `includePath` | string or string[] | Archive-relative include directory. Remote zip archives install as headers only when this is set. |
| `stripPrefix` | string | Treat an archive-relative directory as the source root after extraction. |
| `patches` | string or string[] | Project-relative patch files to apply with `git apply` after extraction and `stripPrefix`. |
| `components` | string or string[] | Copy only selected top-level entries from the include directory or project root. |
| `checksum` | string | Expected SHA-256 archive digest, as a raw digest or `sha256:<digest>`. |

Example with install modifiers:

```json
{
  "dependencies": {
    "vendor-sdk": {
      "source": "https://example.com/downloads/vendor-sdk.zip",
      "checksum": "sha256:0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
      "stripPrefix": "sdk",
      "includePath": ["include", "single_include"],
      "components": ["vendor"],
      "patches": ["patches/vendor-sdk.patch"]
    }
  }
}
```

## How Install Mode Is Selected

This determines whether a dependency ends up in `cpp_libs/include` or `cpp_libs/projects`.

### Default Rules

| Input | Default behavior | Result |
| --- | --- | --- |
| GitHub/Gitee repository with releases | Check whether the release archive has a usable include directory | Use `include` if found, otherwise fall back to `projects` |
| GitHub/Gitee repository without releases | Use the repository source archive | Usually installs into `projects` |
| `--branch` | Skip releases and use the branch source archive | Probe or install from that source archive |
| `--full-project` | Skip include detection | Install directly into `projects` |
| Remote zip | Skip GitHub/Gitee release logic | Install into `projects` by default |
| Remote zip + `--include-path` | Use the selected directory as the include root | Install into `include` |

### Option Order

| Stage | Related fields |
| --- | --- |
| 1. Download archive | `tag`, `branch`, `versionRange`, `versionPolicy`, `prerelease` |
| 2. Verify archive | `checksum` |
| 3. Adjust source root | `stripPrefix` |
| 4. Apply patches | `patches` |
| 5. Choose headers or full project | `fullProject`, `includePath` |
| 6. Restrict copied entries | `components` |
| 7. Write lockfile and metadata | `cppkg-lock.json`, `cpp_libs/deps.json` |

### Update And Remove

| Command | Behavior |
| --- | --- |
| `cppkg-cli update` | Removes recorded old paths, then reinstalls from the recorded source. Without new options, it reuses the previous install mode and version selection. |
| `cppkg-cli remove` | Deletes paths recorded in `deps.json`. Shared paths still referenced by other packages are kept. |
| `cppkg-cli install --frozen-lockfile` | Requires `cppkg-lock.json` to match `cppkg.json`, then installs from the lockfile. |

Downloaded archives are cached in the configured cache directory. Add `--no-cache` to `get`, `install`, or `update` when you need to download again.

## CMake Integration

Generate `cppkg.cmake`:

```bash
cppkg-cli cmake
```

Include it from `CMakeLists.txt`:

```cmake
include("${CMAKE_CURRENT_LIST_DIR}/cppkg.cmake")
target_link_libraries(my_target PRIVATE cppkg::headers)
```

`cppkg::headers` exposes the shared include directory. `cppkg.cmake` also auto-adds full-project installs that contain `CMakeLists.txt`.

## Project Config

Project config is stored in `./cppkg.config.json`:

```bash
cppkg-cli config set proxy http://127.0.0.1:7890
cppkg-cli config set githubToken ghp_xxx
cppkg-cli config set giteeToken xxxxxx
cppkg-cli config set packageRootDir third_party/cppkg
cppkg-cli config set includeDirName include
cppkg-cli config set projectsDirName projects
cppkg-cli config set cacheDirName cache
cppkg-cli config get packageRootDir
cppkg-cli config list
cppkg-cli config remove proxy
```

Supported config keys:

| Key | Default | Meaning |
| --- | --- | --- |
| `proxy` | empty | Default proxy for HTTP and HTTPS requests. |
| `httpProxy` | empty | Default HTTP proxy. |
| `httpsProxy` | empty | Default HTTPS proxy. |
| `githubToken` | empty | Token for GitHub private repositories, release assets, source archives, and search rate limits. |
| `giteeToken` | empty | Token for Gitee private repositories, releases, and source archives. |
| `packageRootDir` | `cpp_libs` | Install data root. |
| `includeDirName` | `include` | Shared include directory name under `packageRootDir`. |
| `projectsDirName` | `projects` | Full-project directory name under `packageRootDir`. |
| `cacheDirName` | `cache` | Download cache directory name under `packageRootDir`. |
| `depsFileName` | `deps.json` | Installed package metadata file name under `packageRootDir`. |

GitHub tokens can also come from `GITHUB_TOKEN` or `GH_TOKEN`; Gitee tokens can also come from `GITEE_TOKEN`. Project config takes precedence over these environment variables.

CLI proxy flags take precedence over config:

```bash
cppkg-cli get https://github.com/nlohmann/json \
  --http-proxy http://127.0.0.1:7890 \
  --https-proxy http://127.0.0.1:7890
```
